# Shopping Item Encryption Implementation Plan

## 1. 目的

家族・サークル向けのQR招待フローにおいて、ショッピングアイテム用の共通鍵を安全に配布し、参加後に復号可能な状態を自動で構築する。

## 2. スコープ

- 対面QRスキャンを起点とした参加フロー
- 受諾者の公開鍵送信
- 招待者による共通鍵暗号化返信
- 受諾者による復号とローカル保存
- UI上の待機表示とエラーハンドリング

## 3. 前提

- Firestoreを鍵交換イベントの同期チャネルとして使用
- ローカル保存は既存の永続化基盤（Hive）を利用
- 鍵マテリアルはBase64文字列としてFirestoreに保存

## 4. アーキテクチャ方針

- 共通鍵: グループ単位の対称鍵（AES想定）
- 鍵配布: 受諾者公開鍵による暗号化配布
- 同期: Firestoreのリアルタイム監視
- 安全性: 平文鍵をネットワーク保存しない

## 5. 実装フェーズ

### Phase 1: 暗号基盤の準備

1. cryptographyパッケージを依存関係に追加する。
2. 暗号ユーティリティを作成する。
3. 共通鍵生成、公開鍵ペア生成、暗号化、復号APIを実装する。
4. 鍵とバイト列のBase64変換ユーティリティを実装する。

成果物:

- 暗号ユーティリティクラス
- 単体テスト（正常系と失敗系）

### Phase 2: Firestoreスキーマ拡張

1. 招待ドキュメントに公開鍵格納フィールドを追加する。
2. 返信用の暗号化済み共通鍵フィールドを追加する。
3. ステータス管理フィールド（待機中、返信済み、完了、失敗）を追加する。
4. 更新時刻を保持し、タイムアウト判定を可能にする。

推奨フィールド例:

- responderPublicKey
- encryptedGroupKey
- keyExchangeStatus
- keyExchangeUpdatedAt

### Phase 3: 招待受諾側（Guest）実装

1. QR読み取り後に一時公開鍵ペアを生成する。
2. responderPublicKeyをFirestoreへ送信する。
3. encryptedGroupKeyの書き戻しをリアルタイム監視する。
4. 受信後に秘密鍵で復号し、共通鍵をローカル保存する。
5. 成功時に参加完了へ遷移し、失敗時は再試行導線を表示する。

### Phase 4: 招待承認側（Host）実装

1. 公開鍵付き受諾イベントを検知する。
2. 対象グループの共通鍵を取得する。
3. responderPublicKeyで共通鍵を暗号化する。
4. encryptedGroupKeyをFirestoreへ返信する。
5. 成功後にステータスを返信済みに更新する。

### Phase 5: UI統合と運用性

1. スキャン後に「セキュア接続を確立中」の待機表示を実装する。
2. タイムアウト、無効鍵、権限不足時のエラーメッセージを定義する。
3. 再試行ボタンとキャンセル導線を実装する。
4. 開発ログと運用ログの最小セットを整理する。

## 6. テスト計画

### 単体テスト

- 鍵生成の妥当性
- 暗号化と復号の往復一致
- 不正鍵や破損データ時の例外処理

### 結合テスト

- 2端末での鍵交換一往復
- オフライン復帰時の整合性
- 重複受諾イベント時の冪等性

### E2Eテスト

- QR読み取りから参加完了までの成功率
- 平均完了時間
- 失敗時の再試行成功率

## 7. 受け入れ条件

- 平文の共通鍵がFirestoreに保存されない。
- QR受諾から復号完了までをアプリ内で完結できる。
- 鍵交換失敗時にユーザーが再試行可能である。
- 主要シナリオのテストが継続実行可能である。

## 8. リスクと対策

- リスク: 公開鍵未到着で待機が長引く。
- 対策: タイムアウトと再送導線を実装。

- リスク: ドキュメント競合による上書き。
- 対策: ステータス遷移を厳格化し、冪等処理にする。

- リスク: 端末差による暗号実装差異。
- 対策: 暗号処理を単一ユーティリティへ集約する。

## 9. 実装順序（推奨）

1. Phase 1（暗号基盤）
2. Phase 2（スキーマ）
3. Phase 3（Guest）
4. Phase 4（Host）
5. Phase 5（UI）
6. テスト強化と運用ログ整備
